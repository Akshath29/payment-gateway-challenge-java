# Payment Gateway Challenge

This repository contains my implementation of the Java payment gateway challenge.

## Setup

- JDK 17
- Docker

Start the bank simulator with:

```bash
docker-compose up
```

The gateway runs on port `8090`.

## API

- `POST /payments` - process a payment
- `GET /payments/{id}` - retrieve a payment by id

OpenAPI / Swagger UI:

http://localhost:8090/swagger-ui/index.html

## Design Decisions

- Currency is modeled as an enum (`GBP`, `USD`, `EUR`) to restrict accepted values and keep validation simple.
- API models are implemented as Java records to reduce boilerplate and keep DTOs immutable.
- Request payload validation is handled using Bean Validation annotations on the request model.
- Expiry date validation is implemented as a class-level validator because month and year are provided as separate fields.
- Payment IDs are generated by the gateway, and successful/declined payments are stored in an in-memory repository.
- Bank responses are mapped as follows: authorized -> `Authorized`, unauthorized -> `Declined`, and bank `5xx` -> `Declined` with HTTP `503` from the gateway.

## Assumptions

- Invalid input is treated as a bad request and returned as HTTP `400`.
- The gateway only stores and returns the last four card digits; the full card number is never returned in responses.
- The payment ID is generated by the gateway and is independent of any bank authorization code.
- For bank unavailability (`5xx`), the gateway returns HTTP `503` and a declined payment response.
- The minimum allowed amount is `0` (as currently enforced by validation).
- The bank simulator is assumed to handle repeated requests idempotently.
